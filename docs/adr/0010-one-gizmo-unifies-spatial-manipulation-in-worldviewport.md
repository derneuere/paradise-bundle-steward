# One gizmo unifies all spatial manipulation in the WorldViewport

The pre-bulk-transform **WorldViewport** has a small zoo of direct-manipulation affordances — per-section drag (cascade-aware), per-corner drag (shared-corner-aware), per-portal-anchor drag (mirror-aware), per-boundary-line endpoint drag, per-trigger-box transform handles, and the schema-driven spatial-walker overlays for any resource whose schema tags fields `spatial: 'vec3'` / `spatial: 'vec2-xz'`. When designing **Bulk transform** we considered keeping these handles for single-entity selection and adding a separate gizmo only for multi-entity bulk (smallest blast radius — two visual languages coexist), versus unifying onto one on-canvas translate/rotate gizmo for any **Selection** at any cardinality. We chose unification: there is exactly one gizmo on screen at any time, anchored at the **Selection**'s **Pivot**, regardless of whether the **Selection** is a single corner, a single section, a mixed bag of trigger boxes and portals, or a cross-**Bundle** stadium. Rationale: two visual languages in the same viewport is the next friction; "general 3D editor UX" means one gizmo in everyone's vocabulary; and consolidating onto one affordance is the only way to make the cardinality-1 and cardinality-N rules uniform (especially the cascade rule from ADR-0009). The cost is real: a substantial refactor of the existing drag-handler code in `src/components/aisections/`, `src/components/triggerdata/`, and the schema-editor spatial walkers, plus the loss of some hand-tuned per-handler behaviours (snap-to-edge during single-section drag, automatic shared-corner drag) which are either reproduced via the cascade modifier or deferred. Inspector numeric fields stay unchanged — they live outside the **WorldViewport** and are not part of this consolidation.
