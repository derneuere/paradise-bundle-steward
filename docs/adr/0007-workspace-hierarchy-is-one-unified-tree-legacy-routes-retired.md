# Workspace hierarchy is one unified tree; legacy thin Schema-driven page routes retired

The pre-Workspace **Workspace editor** rendered two cooperating trees: a `WorkspaceTree` for Bundle / Resource / Instance rows, and a separately-mounted `HierarchyTree` for schema rows under the selected Resource. We considered keeping that split (cheaper change, no legacy disruption) and refactoring into a shared tree primitive. We chose instead a single unified `WorkspaceHierarchy` component owning every row level — Bundle, Resource type, Instance, schema — in one virtualised list, modelled on Unity's hierarchy panel where every node is selectable and toggleable as part of the same hierarchy. **Visibility** (eye icon) stays bounded to the Bundle / Resource type / Instance levels per the existing **Visibility** semantics — schema rows are selectable but not toggleable, since a portal is a sub-shape of its **Resource**, not an independent thing in the scene.

Retiring the 9 thin **Schema-driven page** routes (`/streetdata`, `/aisections`, `/trafficdata`, `/triggerdata`, `/zonelist`, `/challengeList`, `/vehicleList`, `/colors`, `/ice`) follows directly: they were pure single-Bundle scaffolding around `SchemaEditor`, and the **Workspace editor** at `/workspace` now covers the same workflows with the unified hierarchy. The 6 **Bespoke page** routes (Renderable, Texture, AttribSysVault, DeformationSpec, PolygonSoupList, Shader) survive — their custom UX (decoded-mesh preview, texture preview, picker UIs, shader compilation) hasn't been folded into the **Workspace editor**'s inspector yet, and that's separate work. Until those Bespoke routes are themselves retired or refactored, the legacy `HierarchyTree` component and the `useFirstLoadedBundle` / `useFirstLoadedBundleId` hooks survive purely as their plumbing.
