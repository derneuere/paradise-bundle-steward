# Resource version variants are EditorProfiles over a discriminated-union model

The Workspace editor needs to support multiple on-disk layouts of the same resource type — AI Sections has Burnout 5 prototype V4/V6 fixtures alongside the retail v12, TrafficData has v22 vs v44/v45, ZoneList has a Nov-2006 variant, and v12 PR (Paradise Remastered, 64-bit pointer layout) is on the horizon. We considered three shapes for handling this. (a) Keep retail-shaped models with optional sidecar fields (`legacyV4V6?: ...`), so consumers stay v12-coloured and dispatch only when a sidecar is present. (b) Promote the model to a discriminated union (`{ kind: 'v12', ... } | { kind: 'v6', ... } | { kind: 'v4', ... }`) and force every consumer to narrow on `kind`. (c) Mint one resource type per variant in the core registry, so `aiSectionsV12` and `aiSectionsV4` are completely separate handlers.

We chose (b) and added a sibling editor-aware registry layer in `src/lib/editor/`. The discriminated union forces narrowing — the writer dispatches on `kind` rather than peeking sidecars, so a v6 model can never accidentally land in the v12 codepath silently. The new `EditorProfile<M>` shape pairs a `kind` discriminator with a schema, an optional 3D overlay, an optional extension registry, and an optional conversions map; the editor registry exposes `pickProfile(typeId, model)` and `profilesFor(typeId)` so the Workspace asks "which schema/overlay should I use for this model?" exactly once at the boundary, and the rest of the editor speaks the narrowed variant directly. This is α-loose layering: the **core** registry stays React-free (consumed by the CLI, the bundle writer, and cross-platform export), and the **editor** registry sits on top of it adding schema/overlay/extension/conversion wiring without polluting `src/lib/core/` with React imports. Read-only EditorProfiles (planned for the V4/V6 next slice) express their read-only-ness through the schema's existing `readOnly` / `addable` / `removable` / `hidden` metadata via the new `freezeSchema()` helper — there is no parallel `editable: false` cap flag at the EditorProfile / handler / page layer, because editability is already a schema concern (per the project's pet-peeve rule in `CLAUDE.md`). The first slice ships only the V12 retail profile; V4/V6 prototype payloads parse and round-trip via the core registry but the schema editor sees them as opaque until their (frozen, read-only) profile lands.

Option (a) was the cheapest immediate change — and the working tree had already started down that path with `legacyV4V6?: LegacyAISectionsData` on `ParsedAISections`. We rejected it because every v12 consumer would have to remember to check the sidecar; one missed check and a v6 fixture renders empty in the editor (or worse, crashes during a `sections.length` read). The discriminated-union check is a structural failure (TypeScript narrows or the codepath has nowhere to read fields from), not a runtime convention. Option (c) — separate handlers per variant — would force every shared concern (BTTSTREETDATA byte-roundtrip tests, the cross-platform export's `getExportablePlatforms`, the registry's typeId↔handler lookup) to know about the variant axis too, and the parser would no longer be the canonical place to choose layouts: callers would have to pre-detect the version themselves. Keeping one core handler per typeId and pushing the variant axis up into the editor registry localises the complexity to where it actually matters — the editor — and preserves the existing one-handler-per-typeId invariant the CLI / bundle writer / export pipeline depends on.

The migration is byte-exact and behaviour-preserving for V12: existing fixtures (`example/AI.DAT`, `example/ps3/AI.DAT`, `example/older builds/AI.dat`) still round-trip identically through the registry's auto-generated test harness, every workspace component test passes unchanged, and the `WorkspacePage` / `ViewportPane` / `WorldViewportComposition` migration replaces three per-key switches (`EXTENSIONS_BY_KEY`, `getSchemaByKey`, the overlay `switch` statement in the composition's `renderOverlay`) with single `pickProfileByKey(key, model)` calls. The next slice — adding a frozen V4/V6 schema and registering V4/V6 EditorProfiles alongside v12 — drops in via `ENTRIES.push(...)` in `src/lib/editor/registry.ts` and a new schema file; no existing call site needs to change.
