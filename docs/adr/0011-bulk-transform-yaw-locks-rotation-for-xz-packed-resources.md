# Bulk transform yaw-locks rotation when the Selection contains XZ-packed resources

Several **WorldViewport** resource families store spatial data as 2D-on-XZ rather than full 3D: AI section corners (`Vector2`), AI section boundary and no-go lines (`Vector4` packed as `startX/startZ/endX/endZ`), zone list points (`Vec2Padded`), and traffic junction/light boxes (`Vector4` with yaw packed in `.w`). None of these have a Y component to receive a pitch (around X) or roll (around Z) rotation — applying one would either be silently lossy or require a file-format-aware data migration to promote every XZ-packed type to 3D. Steward's BND2 byte-for-byte writeback is an invariant (CONTEXT.md), so a wholesale type promotion is off the table. We considered three options: yaw-only globally (matches the data uniformly but feels constrained for the trigger-box / static-vehicle case which actually have full 3-axis); full 3-axis with silent ignore on XZ-locked types (incoherent UX — "I rotated 30° around X and the boundary lines didn't move"); full 3-axis with **auto-disabled** X and Z rings whenever any XZ-locked resource is in the **Selection**. We chose the third. The gizmo always renders three rotation rings so the affordance feels like a 3D editor; the X and Z rings grey out and become non-interactive when the **Selection** contains any XZ-packed resource. **Selections** that contain only 3D-capable resources (trigger box regions with their Euler rotation, static traffic vehicles with their `Matrix44`, portal anchors as pure points, street-data reference positions) get the full three-axis ring set. Burnout Paradise's world is fundamentally horizontal, so the typical "rotate the stadium" gesture is yaw — the auto-disable is invisible to most workflows and only surfaces when a user genuinely wants to tilt geometry that the data model cannot represent.
