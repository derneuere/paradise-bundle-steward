# WorldViewport overlays receive resource data as React props, not from context

We considered making each overlay a self-contained value that fetches its own resource from `BundleContext` — under that shape, multi-overlay routes would compose "for free." We chose instead to keep overlays as React components receiving `(data, selectedPath, onSelect)` as props, with the page (and future EditorRoute) responsible for fetching data from `BundleContext` and threading it down. This keeps overlays testable in isolation with hand-crafted fixture data (no context provider needed in tests), keeps "what is being edited" cleanly separate from "how it is rendered," and still supports multi-overlay routes via plain React children composition — `<WorldViewport><AIOverlay …/><CollisionOverlay …/></WorldViewport>`.
