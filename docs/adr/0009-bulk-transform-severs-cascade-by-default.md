# Bulk transform severs cascade by default

The legacy single-section drag in the **WorldViewport** uses `translateSectionWithLinks` (in `src/lib/core/aiSectionsOps.ts`), which automatically cascades a translate to neighbouring sections — dragging their reverse-portal anchors and any corners that lie on the shared edge. That's right for nudging a section by a few metres, but for the use case the new **Bulk transform** is built around — relocating a whole island like the stadium to the other side of the map and rotating it — cascade produces stretched portal anchors and partial deformation on every section that links into the bulk from outside. We considered three options: (a) keep cascade on for single-section drag, off for bulk (cleaves the model at cardinality 2 and is hard to explain); (b) cascade always on by default with a modifier to sever (the inverse of what the use case wants); (c) cascade always off by default with a modifier to enable. We chose (c) — applied uniformly to single-entity gestures via the unified gizmo (ADR-0010) and to multi-entity bulks. The mental model is one sentence: "the gizmo moves what you selected; modifier extends to connected geometry." The cost is a small UX shift for existing single-section drag (no longer auto-cascades) and the introduction of "stale" mirror-portal anchors after a bulk relocation — a portal still links by `linkSection` ID but its position is out of sync with its mirror across the boundary. We accept this: surfacing dangling boundary portals for re-wiring is a discoverable follow-up affordance (highlight / inspector list), and not a v1 blocker. The legacy `translateSectionWithLinks` function stays in the codebase as the modifier-on path.
